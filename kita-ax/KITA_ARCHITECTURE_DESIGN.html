아키텍처 설계 원칙
(Architecture Principles)
1. 보안 내재화 (Security by Design)

보안을 사후 적용이 아닌 아키텍처 수준에서 기본 요소로 내재화. 모든 데이터 흐름에 보안 검증 레이어가 반드시 거친다.

2. 최소 권한 원칙 (Principle of Least Privilege)

모든 주체(사용자, 에이전트, 시스템)는 필요한 최소한의 권한만 동적으로 부여받는다. 초과 권한은 허용하지 않는다.

3. Zero Trust 모델 (Never Trust, Always Verify)

네트워크 위치나 이전 인증 여부와 관계없이, 모든 접근 요청마다 속성과 정책을 재검증한다. 특히 에이전트의 API 호출은 매번 새로운 인증서를 발급받는다.

4. 정책과 코드의 분리 (Separation of Concerns)

RBAC/ABAC 정책을 비즈니스 로직 코드와 물리적으로 분리하여, 정책 변경이 개발 배포 사이클과 독립적으로 가능하도록 한다. (OPA 기반)

5. 감시 가능성 (Observability)

모든 보안 관련 이벤트(인증, 인가, 데이터 접근, API 호출, 마스킹)는 타임스탬프와 함께 중앙 로그에 기록되어 사후 감사가 완벽히 추적 가능해야 한다.

6. 이기종 환경 지원 (Heterogeneous Environment Support)

DB, 파일, 벡터 DB, API 등 다양한 데이터 소스와 온프레미스/클라우드 인프라를 유기적으로 통합하되, 보안 정책은 일관되게 적용된다.

Layer 1: AI 가드레일 정의 레이어
(AI Guardrail Definition Layer - SER-AP-001)
목적 (Objective)

AI 에이전트가 안전하고 일관되게 행동하도록 정책 기반 필터링과 윤리 검증을 양방향으로 적용한다. 사용자 입력이 LLM에 도달하기 전과 LLM이 답변을 반환한 후 모두 검증한다.

입력 필터 (Inbound Interceptor)
검증 단계 처리 내용 기술
1. PII 탐지 & 마스킹 개인정보(주민번호, 전화, 이메일) 자동 탐지 및 [PII-ID] 토큰으로 치환 spaCy NER + 정규표현식
2. 입력값 검증 입력 길이, 포맷, 언어 범위 검증 Custom Validator
3. 프롬프트 인젝션 방어 "ignore previous instructions" 같은 인젝션 패턴 탐지 및 차단 시그니처 기반 필터
4. 윤리 검증 비속어, 욕설, 차별 표현 탐지 토큰 필터링
출력 필터 (Outbound Interceptor)
검증 단계 처리 내용 기술
1. 응답 내 민감정보 재검증 LLM이 답변 중 원래 마스킹된 PII를 복원하려 시도하는 경우 차단 PII 재검사
2. 악의적 컨텐츠 탐지 답변에 포함된 자바스크립트, SQL 인젝션 코드 등 탐지 AST 분석
3. 외부 유출 조항 검증 KITA 연계 데이터가 응답에 포함되었는지 확인. 포함 시 해당 부분 마스킹 데이터 태그 매칭
4. 환각 방지 답변이 학습 데이터 범위를 벗어나는 임의 생성 값을 포함하는지 검증 근거 문서 매칭
핵심: 모든 입출력은 "LLM 모델에 데이터가 도달하기 전" 그리고 "사용자에게 응답이 반환되기 전" 반드시 이 필터를 통과해야 한다. 우회는 불가능하다.
Layer 2: 데이터 접근 제어 레이어
(Data Access & Isolation Layer - SER-AP-002)
1차 방어선: RBAC (Role-Based Access Control)

목적: 에이전트 실행 권한 및 기능 진입을 역할(Role) 기반으로 제어. 사용자가 특정 에이전트 자체를 실행할 수 없으면 데이터 접근도 발생하지 않는다.

역할 가능한 에이전트 데이터 접근 범위 관리 권한
시스템 관리자 전체 전체 (제약 없음) 정책 관리, 로그 조회
KITA 직원 내부 전용 에이전트 부서별 귀속 데이터 없음
일반 회원 (기업) 무역 지원, 관세 조회 자사 데이터만 없음
관세사 통관 대행, 세율 계산 위임받은 기업 데이터 없음
감사자 (Auditor) 없음 (실행 불가) 로그 읽기 전용 감사 로그 조회
2차 방어선: ABAC (Attribute-Based Access Control)

목적: 실제 데이터 자산 접근을 속성(Attribute) 기반 정책으로 제어. RBAC가 "이 에이전트를 실행할 수 있는가?"를 판단한다면, ABAC는 "이 데이터를 접근할 수 있는가?"를 실시간 검증한다.

ABAC 동작 원리:
에이전트 쿼리 발생 → ABAC 정책 엔진(OPA) 검증 → 주체 속성(Company_ID) == 데이터 속성(Owner_Company_ID) ∧ 환경 속성(업무 시간, 데이터 분류도) 확인 → 조건 일치 시 데이터 반환, 불일치 시 차단 또는 마스킹
데이터 형태별 ABAC 구현

(1) 구조화 데이터 (RDB / PostgreSQL)

  • 필터링 방식: 동적 WHERE 절 주입
  • 구현: Spring Data JPA Interceptor가 쿼리 생성 직전, 현재 사용자의 Company_ID를 자동으로 WHERE 조건에 추가
    • 예: `SELECT * FROM trade_data` → `SELECT * FROM trade_data WHERE company_id = :userCompanyId`
  • 이점: 개발자가 권한 체크 로직을 따로 짤 필요 없음. ORM이 자동 처리

(2) 반구조 데이터 (벡터 DB / PostgreSQL pgvector)

  • 필터링 방식: 메타데이터 Payload Filtering
  • 구현: 벡터 데이터가 저장될 때 `owner_company_id`, `classification_level` 같은 메타데이터를 함께 저장. RAG 검색 시 유사도 검색(Semantic Search)과 속성 필터링을 함께 실행
    • PostgreSQL pgvector: `WHERE owner_company_id = :userCompanyId AND classification_level <= :clearance` 조건을 임베딩 검색에 병합
  • 이점: 의미론적 관련성(유사도)과 접근 제어를 동시에 만족

(3) 비구조 데이터 (파일 / Object Storage)

  • 필터링 방식: 오브젝트 메타데이터 태깅
  • 구현: 파일이 업로드될 때 확장 속성(Extended Attributes)에 `owner`, `classification`, `source_system` 태그 저장. 에이전트가 RAG를 위해 텍스트 청크를 추출하기 전, 이 메타데이터 태그를 먼저 검증
    • AWS S3: Object Tagging으로 `owner:KITA_001`, `classification:internal` 저장
  • 이점: 파일 내용 자체는 건드리지 않고 메타데이터만으로 접근 제어
핵심 설계 원칙: 비즈니스 로직 코드에 권한 체크를 섞지 말 것. 대신 "중앙 정책 엔진(OPA)"이 모든 데이터 쿼리/접근 직전에 정책을 자동으로 끼워 넣는다. 이를 통해 개발 복잡도를 낮추고 정책 변경의 유연성을 확보한다.
Layer 3: 민감정보 비식별화 레이어
(Sensitive Data Masking Layer - SER-AP-003)
목적

LLM 모델이 실제 민감정보(PII, 기업 기밀)에 접근하지 못하도록, Layer 1의 입력 필터에서 민감정보를 자동으로 탐지하고 토큰화하여 치환한다. 원본 데이터는 별도 저장소에만 보관되며, LLM은 토큰 형태의 대체값만 본다.

PII 탐지 카테고리
카테고리 탐지 항목 마스킹 형식 탐지 기술
개인정보 성명 [PII-NAME-001] NER (Named Entity Recognition)
주민등록번호 (RRSSN) [PII-RRSSN-001] 정규표현식: \d{6}-[1234]\d{6}
전화번호 [PII-PHONE-001] 정규표현식: 010-\d{4}-\d{4}
이메일 [PII-EMAIL-001] 정규표현식: \S+@\S+\.\S+
기업정보 사업자번호 [PII-BIZID-001] 정규표현식: \d{3}-\d{2}-\d{5}
매출액 (숫자 큰 금액) [PII-REVENUE-001] 패턴 분석: 10억 이상 숫자
통관 번호 [PII-CUSTOMS-001] 정규표현식: [A-Z]{2}\d{12}
금융정보 계좌번호 [PII-ACCOUNT-001] 정규표현식 + 길이 검증
카드번호 [PII-CARD-001] Luhn 알고리즘
마스킹 전략 및 복원 메커니즘

1. 토큰 생성 및 저장

  • 입력 데이터에서 PII 탐지 → UUID 생성 (예: `[PII-RRSSN-a3f2b1c5-4e8d-...]`)
  • 원본 PII ↔ 토큰 매핑을 별도 암호화 저장소에 보관 (기본 DB와 분리)
  • LLM은 토큰 형태의 마스킹된 데이터만 입력받음

2. 응답 단계 복원

  • LLM이 답변 생성 (토큰 포함 가능)
  • Layer 1의 출력 필터가 답변의 모든 토큰을 스캔
  • 토큰이 원래 입력에서 온 것인지 LLM이 임의 생성한 것인지 검증
  • 인가된 사용자(권한 있음)에게만 토큰을 원본 PII로 복원하여 반환
  • 미인가 사용자에게는 토큰 그대로 또는 추가 마스킹 후 반환
원본 보호 메커니즘: 마스킹되지 않은 원본 데이터가 LLM 모델에 직접 입력되는 것을 원천 차단한다. 모든 PII는 사전에 탐지되고 토큰화되어야만 LLM으로 전달된다. 만약 마스킹 엔진이 PII를 탐지하지 못하고 원본이 유출된 경우를 대비해 별도의 post-check 메커니즘(LLM 응답 스캐닝)을 운영한다.
Layer 4: Agent 시큐어 코딩 레이어
(Agent Secure Coding & Ops Layer - SER-AP-004)
1. 프롬프트 인젝션 방어

아키텍처: 시스템 프롬프트와 사용자 입력을 물리적으로 분리

영역 내용 보호 방식
System Prompt
(격리)		 "You are a trade assistant. Only use these tools: [Tool A, Tool B, ...]" 코드 상수로 고정. 런타임에 변경 불가
User Input
(격리)		 사용자가 입력한 자연어 질문 XML 태그로 명시적 구분
<user_input>...</user_input>
Tool Definitions
(격리)		 Tool A: API endpoint
Tool B: Data query spec		 별도 JSON Schema로 정의. 동적 추가 불가

방어 로직:

  • 사용자 입력에서 "ignore previous", "new instruction" 같은 키워드 탐지 → 해당 입력 차단
  • LLM 응답 검증: 정의되지 않은 Tool 호출 시도 감지 → 차단
  • 재귀적 프롬프트 생성 방지: 에이전트가 생성한 중간 프롬프트도 같은 필터 통과
2. Zero Trust + 최소 권한 (Tool/API 호출 제어)

기본 원칙: 에이전트가 외부 Tool이나 API를 호출할 때마다 매번 새로운 인증서/토큰을 발급받는다. 일회용 자격증명 모델.

단계 처리 기술
1. 요청 인증 에이전트가 Tool 호출 → API Gateway가 요청 인터셉트 → Vault에 토큰 발급 요청 Vault Dynamic Secrets
2. 정책 검증 OPA가 에이전트의 역할과 현재 요청된 Tool을 매칭 → 권한 있는 Tool만 실행 승인 OPA Policy Engine
3. 토큰 발급 Vault가 해당 Tool 전용 API 키를 동적 생성 (유효 시간: 5분) Vault Leasing
4. Tool 호출 에이전트가 발급받은 토큰으로 API 호출 OAuth 2.0 Bearer Token
5. 로깅 & 감시 모든 API 호출 기록 (타임스탬프, 호출자, Tool, 파라미터, 응답 상태) OpenSearch + Logstash
6. 토큰 자동 만료 유효 시간 만료 → Vault가 자동 폐기. 재발급 필요 Vault TTL
최소 권한 원칙 적용 예:
에이전트가 "수출입 통계 조회" Tool만 필요하면, 그 Tool 전용의 읽기 API 키만 발급. 데이터 삭제, 쓰기, 관리자 권한은 절대 포함하지 않음.

Layer 5: Hybrid Multi-LLM 오케스트레이션
(Hybrid Multi-LLM Orchestration)
목적

모든 요청이 4개 보안 레이어를 통과한 후, LLM 모델로 라우팅할 때는 "민감정보 포함 여부"와 "필요한 계산 복잡도"에 따라 SaaS LLM (성능) 또는 Private LLM (보안) 중 선택적으로 라우팅한다. 사용자 관점에서는 투명하지만, 보안과 비용 최적화를 동시에 확보한다.

격리된 보안 존
(VPC 분리)

분류 기준 (상세):

  • 민감정보 탐지: Layer 3의 마스킹 엔진이 PII/기업 기밀을 탐지했으면 "민감" 판정
  • 계산 복잡도: 요청 토큰 길이, 쿼리 복잡도, 문맥 윈도우 크기 등을 기준으로 판단
  • 데이터 소유권: KITA 연계 서비스 데이터(외부 유출 금지)가 포함되면 Private LLM으로 강제
SaaS LLM 사용 시 보안 조항
  • "no_training": true 옵션을 API 요청에 명시 → OpenAI/Anthropic이 해당 대화를 학습 데이터로 사용하지 않음
  • API 요청 본문에 민감정보(마스킹된 토큰) 포함 가능하지만, 원본 PII는 절대 포함 금지
  • API 응답은 Layer 1의 출력 필터를 재통과
Private LLM 운영
  • 모델: Llama 3 (70B) 또는 Mistral (7B/34B)
  • 배포: Ollama를 통해 클라우드 내 독립 컨테이너로 실행
  • 네트워크 격리: 별도 VPC, 인터넷 접근 불가
  • 성능: SaaS 모델보다는 낮지만 기업 데이터 처리에는 충분
기대 효과: 민감하지 않은 일반 질문(예: 관세율 일반 설명)은 SaaS의 고급 모델로 빠르게 처리하고, 기업 기밀이 포함된 질문(예: "우리 회사 매출액에 따른 관세"는 온프레미스 Private LLM에서만 실행. 사용자는 차이를 느끼지 못하면서 보안과 성능을 동시에 확보.

배포 구성 및 인프라
(Deployment & Infrastructure)
아키텍처 배포 환경

기본 전제: 클라우드 기반 (AWS / Azure / GCP 중립)

  • 메인 애플리케이션: 클라우드 VPC 내 컨테이너 (Kubernetes 권장)
  • Private LLM: 격리된 VPC / Subnet (인터넷 미노출)
  • 데이터 저장소: RDS (PostgreSQL + pgvector) 또는 관리형 데이터베이스
  • 로깅 인프라: OpenSearch Domain (AWS OpenSearch / Elasticsearch 호스팅)
온프레미스 연동 (Hybrid 지원)
  • KITA 기존 온프레미스 시스템 데이터: AWS PrivateLink 또는 VPN을 통해 보안 연결
  • 데이터 동기화: 배치 프로세스 (nightly) 또는 실시간 스트리밍 (Kafka)
  • 네트워크 보안: WAF (Web Application Firewall) + Security Group 설정
