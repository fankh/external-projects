## 🎯 대응 핵심 메시지

### 질문 1: "Agentic AI를 RBAC/ABAC로 어떻게 보호하나?"

**답변 틀:**

#### 1.1 RBAC (역할 기반 제어) — Guardrail의 제1방어선

**에이전트의 역할 기반 권한 제어:**
```
Agent 요청 (예: "문서 읽기")
    ↓
JWT 토큰에서 Agent의 역할 추출
    ↓ (예: "power_user")
OPA Policy Engine에서 정책 평가
    ↓
role: power_user + resource: document:confidential → DENY
    ↓
요청 거부 + 감사로그 기록
```

**KYRA의 RBAC 5단계 역할:**

| 역할 | 공개 문서 | 내부 문서 | 기밀 문서 | Agent Tool 호출 |
|---|---|---|---|---|
| **Admin** | ✓ | ✓ | ✓ | 전체 Tool 사용 |
| **Manager** | ✓ | ✓ | ✓ | 제한된 Tool만 |
| **Power User** | ✓ | ✓ | ✗ | 지정된 Tool만 |
| **User** | ✓ | ✓ | ✗ | 읽기 전용 Tool |
| **Viewer** | ✓ | ✗ | ✗ | 도구 호출 불가 |

**메시지:** "Agent가 어떤 역할로 실행되는지에 따라 접근 범위를 사전에 제한. 무역 Agent도 Manager 역할로 설정하면 기밀 문서 자동 차단됩니다."

#### 1.2 ABAC (속성 기반 제어) — 세분화된 동적 정책

**Agent 요청 시 동시에 평가되는 4가지 속성:**

1. **`doc.classification`** — 문서 민감도
   - public | internal | confidential | restricted

2. **`user.role`** — 사용자 역할
   - admin | manager | user | viewer

3. **`user.tenant_id`** — 테넌트 격리
   - 기업 간 데이터 100% 분리 (무역사 A ↔ 무역사 B 차단)

4. **`agent.action`** — 에이전트 동작
   - read | write | tool_call | export

**동적 정책 예시 (KITA 시나리오):**
```rego
# "restricted 문서는 admin만 읽기 가능"
allow_read {
    doc.classification == "restricted"
    user.role == "admin"
}
부정일 경우: DENY + 감사로그
```

**추가 안전장치:**
- **비식별화 (Masking):** ALLOW 시에도 PII 자동 마스킹
  - 통관번호, 사업자번호, 계좌번호 자동 가림 (****-****-****)
- **감사로깅:** 모든 접근 시도 기록 (성공/실패)
- **정책 변경 실시간 반영:** OPA에서 정책 수정 → 5ms 이내 모든 Agent에 적용

**메시지:** "단순 역할뿐 아니라, 문서의 민감도, 사용자 부서, 테넌트, 시간대까지 고려한 다차원 정책으로 오탐 거의 0에 가까움. 기밀 정보도 필요한 사람은 즉시 승인 가능."

#### 1.3 제어 체크포인트 (Layer 3 & 4)

**요청이 지나가는 5개 검사소:**

1. **Gateway (Kong)** → JWT 유효성, 시그니처 검증
2. **Auth Service** → 사용자/Agent 신원 확인
3. **OPA Policy Engine** → RBAC/ABAC 규칙 평가 ← **KYRA의 핵심**
4. **RAG Service** → 문서 조회 + PII 마스킹
5. **Audit Logger** → 모든 접근 기록

**메시지:** "각 요청이 5개 검사소를 모두 통과해야만 데이터 반환. 특히 OPA에서 정책 위반 시 즉시 차단되고, 어떤 Agent도 정책을 우회할 수 없는 구조입니다."

### 질문 2: "구축 시 솔루션 + 커스터마이징을 어떻게 병행하나?"

**답변 틀:**

#### 2.1 3단계 구축 방식

```
┌─────────────────────────────────────────────────────────────┐
│                     6개월 구축 로드맵                         │
├────────────────────┬─────────────────────┬──────────────────┤
│   Phase 1          │    Phase 2          │    Phase 3       │
│  (1~2개월)         │   (3~5개월)         │   (6개월~)       │
├────────────────────┼─────────────────────┼──────────────────┤
│ ⚡ 솔루션 기본    │ 🔧 맞춤 커스터마이징│ ✅ 운영 안정화   │
└────────────────────┴─────────────────────┴──────────────────┘
```

**Phase 1 — 솔루션 기본 도입 (1~2개월)**
- Docker Compose로 33개 컨테이너 배포 (pre-built)
- RBAC/ABAC 기본 정책 활성화
- RAG 시스템 연동 (Milvus, MinIO, Redis)
- JWT 기반 Multi-Tenancy 활성화
- **결과:** 즉시 사용 가능한 AI Guardrail 완성
- **투입:** SeekersLab 엔지니어 2~3명 (배포, 기초 교육)

**Phase 2 — KITA 맞춤 커스터마이징 (3~5개월)**
- **도메인 학습:** 무역 데이터로 RAG 재학습
  - 통관 절차, 관세 분류, 수출입 규제 등 무역 용어 학습
  - 기존 Milvus 벡터DB + 무역 문서 인제스션
  
- **PII 탐지 규칙 확대:**
  - 기본: 주민번호, 이메일, 휴대폰 번호
  - 확대: 통관번호(B/L), 사업자번호, 은행 계좌번호
  - 선적지(Country Code), 위험국가 자동 감지
  
- **Agent Tool 권한 정책 설계:**
  - 무역 Agent의 실행 Tool 화이트리스트 구성
    - ✓ 관세청 API 조회 (조회만 가능, 수정 불가)
    - ✓ 환율 조회 서비스
    - ✗ 결제 API (접근 차단)
  - 워크플로우별 정책: 승인 필요 여부 등
  
- **대시보드 커스터마이징:**
  - KITA 대시보드에 AI Guardrail 정책 관리 탭 추가
  - 무역 메트릭스 추가 (차단 건수, PII 마스킹 통계 등)
  
- **투입:** SeekersLab 엔지니어 3~4명 (상주 지원 가능)

**Phase 3 — 운영 안정화 + 이관 (6개월~)**
- **성능 최적화:**
  - Latency <200ms (목표), Block Rate >99.5%
  - 정책 엔진 성능 튜닝 (OPA → Rego 쿼리 최적화)
  
- **운영 문서화:**
  - AI Guardrail 운영 매뉴얼 (한국어)
  - 보안 정책서 (정책 수정 프로세스, 감시 기준 등)
  - 트러블슈팅 가이드
  
- **KT DS 운영팀 기술 이전:**
  - 3~4회 기술 교육 (정책 수정, 모니터링, 긴급 대응)
  - 온사이트 지원 종료, 원격 지원으로 전환
  
- **투입:** SeekersLab 엔지니어 1~2명 (최종 검증, 교육)

#### 2.2 비용 효율성

**기존 방식 vs KYRA 솔루션 + 커스터마이징:**

| 항목 | Full Custom Build | KYRA 솔루션 |
|---|---|---|
| **개발 기간** | 12개월+ | 6개월 |
| **초기 투입 인력** | 8~10명 | 3~4명 |
| **관리 공수** | 월 200시간+ | 월 30시간 |
| **예상 비용** | ₩3~5억 | ₩1.5~2.5억 (33% 절감) |
| **위험도** | 높음 (신규 개발) | 낮음 (검증된 제품) |

**메시지:** "KYRA는 이미 검증된 AI Guardrail 플랫폼이므로, Phase 1에서 기본 기능을 2개월 만에 가동하고, Phase 2에서만 무역 도메인에 맞게 커스터마이징. Full Custom Build보다 개발 기간 50% 단축, 비용 33% 절감."

#### 2.3 상주 지원 모델

**개발 단계별 지원:**
- **Phase 1:** 온사이트 2주 (배포 + 초기 설정)
- **Phase 2:** 상주 지원 가능 (요청 시)
  - 주 4일 근무, 원격 협업 1일
  - 실시간 요구사항 반영, 통합 테스트 주도
  - KITA 팀과 pair programming
- **Phase 3:** 부분 상주 (주 2일)

**메시지:** "해외 제품과 달리, 로컬 팀의 상주 지원으로 언어/문화 차이 없이 신속한 소통 가능. KITA의 요구사항을 그 자리에서 확인하고 반영할 수 있습니다."

## 🎤 미팅 시나리오 Q&A

### Q1: "RBAC만으로는 안 되나? ABAC가 정말 필요한가?"

**A:** "RBAC는 제한적입니다. Agent를 'Manager' 역할로 주면, Manager가 접근할 수 있는 모든 내부 문서에 접근 가능. 그런데 무역사의 경우, Manager 역할이라도 특정 프로젝트나 거래처의 기밀 문서는 차단해야 하거든요.

ABAC를 추가하면:
- `doc.classification` + `doc.project_id` + `user.team_id` 등을 동시에 평가
- Manager 역할이라도 '자신의 팀 프로젝트'만 접근 가능
- 다른 팀의 기밀 문서는 자동 차단

이렇게 하면 권한 오설정으로 인한 정보 유출을 거의 100% 막을 수 있습니다."

### Q2: "Phase 2에서 5개월은 긴데, 단축 가능한가?"

**A:** "Phase 1 완료 후 무역 데이터 준비 상태에 따라 달라집니다:

- RAG 학습 데이터가 즉시 준비되면 → 3개월으로 단축 가능
- PII 탐지 규칙을 단계적으로 확대하면 → 병렬 처리로 1~2주 단축

다만 '품질'이 중요한데, 서둘러서 정책이 잘못 설정되면 나중에 더 큰 문제가 됩니다. 차라리 Phase 2를 충실히 하고, Phase 3부터 운영 자동화(모니터링, 자동 업데이트)를 도입하는 게 낫습니다."

### Q3: "다른 AI Guardrail 제품과의 차이점은?"

**A:** "주요 3가지:

1. **KITA 무역 도메인 최적화:** 통관번호, 관세 분류 등 무역 특화 PII 탐지
2. **한국 팀의 상주 지원:** Lakera(미국) 같은 해외 제품과 달리 현지 지원 가능
3. **RBAC + ABAC 동시 적용:** 대부분 RBAC만 지원하는데, KYRA는 정책 엔진(OPA)으로 ABAC까지 자유롭게 구성

특히 '정책을 계약 후 언제든 수정할 수 있다'는 게 큰 차이. 해외 제품은 수정 요청 후 몇 주 걸리지만, KYRA는 OPA 정책만 수정하면 5분 내 반영됩니다."

### Q4: "보안 정책서는 누가 작성하나?"

**A:** "Phase 2와 Phase 3에서:

- **Phase 2:** SeekersLab이 기술 정책서 초안 작성 (RBAC/ABAC 정책, 기술 요구사항)
- **Phase 3:** KT DS와 함께 최종 정책서 검수 및 승인

운영 정책서(SOP)는 KT DS 운영팀이 작성하되, SeekersLab이 템플릿과 기술 배경 설명 제공. 대면 미팅 후 정책서 작성 가이드라인을 함께 공유합니다."
