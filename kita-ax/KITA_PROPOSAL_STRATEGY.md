# KITA 차세대 무역플랫폼 구축 사업 - AI 가드레일 & 보안 제안 전략

**프로젝트:** 한국무역협회(KITA) 차세대 무역플랫폼 구축 사업 1단계  
**담당 파트:** AI 서비스 및 LLM 에이전트 보안 구축  
**버전:** v1.0  
**작성일:** 2026-05-21

---

## 📋 Executive Summary

KITA 사업의 핵심 요구사항은 **"엔터프라이즈 환경에서 AI/LLM 에이전트를 안전하게 운영하기 위한 통합 보안 가드레일 프레임워크"** 구축이다. 

우리(SeekersLab)의 **AI Sandbox** 및 **KYRA 플랫폼** 콘셉트를 이와 정확하게 매칭하여, 단순한 기능 구현이 아닌 **"보안이 내재화된 엔터프라이즈급 LLM 운영 아키텍처"** 를 제안함으로써 압도적인 기술 우위를 확보할 수 있다.

## 🎯 Part 1: RFP 핵심 요구사항 분석

### 1.1 우리가 담당하는 4가지 핵심 보안 요구사항 (SER-AP-001 ~ 004)

#### **SER-AP-001: AI 가드레일 정의 (AI Guardrail Definition)**
- **목적:** AI 에이전트가 안전하고 일관되게 행동할 수 있도록 공통 정책 수립
- **세부 요구사항:**
  - 행동 정책: 시나리오/워크플로우 기반 작동, 매뉴얼/템플릿 답변 구조화, 근거 자료 제공
  - 보안: 민감 데이터(개인정보, 매출정보) 사전 비식별화, 데이터 가공 시 유출 방지, 비인가 외부 서비스 접근 차단
  - 윤리: 비속어/욕설 차단, 환각(Hallucination) 방지, LLM 공급사 윤리 정책 준수

#### **SER-AP-002: LLM 모델 데이터 접근 보안 (Data Access & Isolation)**
- **목적:** 통합 데이터 플랫폼의 학습 데이터 유출 원천 차단
- **세부 요구사항:**
  - 학습 데이터 유출 방지: KITA 연계 서비스 데이터가 외부 LLM 학습에 사용되지 않도록 보호
  - 임의 재가공 방지: 에이전트 권한 밖의 데이터 수정/삭제 금지
  - 외부 유출 방지: 작업 데이터 및 대화 이력의 임의 전송·저장 차단

#### **SER-AP-003: 민감정보 비식별화 (Sensitive Data Masking)**
- **목적:** LLM 모델 입력 전 단계에서 PII(개인정보) 및 기업 기밀 자동 보호
- **세부 요구사항:**
  - 민감정보 자동 탐지 및 마스킹 기능
  - 비식별화 미처리 원본 데이터의 LLM 직접 입력 방지

#### **SER-AP-004: Agent 보안 취약점 강화 (Secure Coding & Ops)**
- **목적:** 프롬프트 인젝션, 무단 API 호출 등 시스템적 공격 방어
- **세부 요구사항:**
  - Tool 호출 제어: Zero Trust + 최소 권한 원칙, Vault 기반 동적 인증, API 호출 이력 로깅
  - 프롬프트 인젝션 방어: 시스템 프롬프트 캡슐화, 입력값 검증·필터링
  - 결과 검증: 출력값 민감정보 마스킹, 악의적 스크립트 차단, 전체 이력 로그 기록

### 1.2 RFP에서 명시하지 않은 것 (우리가 제안해야 할 부분)

### 2.2 기술 제안 구조 (4+1 Layer Architecture)

```
┌─────────────────────────────────────────────────────────────┐
│  1️⃣  AI Guardrail Definition Layer (SER-AP-001)            │
│      → 가드레일 정책 + 행동 규칙 엔진                          │
├─────────────────────────────────────────────────────────────┤
│  2️⃣  Data Access & Isolation Layer (SER-AP-002)             │
│      → RBAC (기능) + ABAC (데이터) 듀얼 제어                 │
├─────────────────────────────────────────────────────────────┤
│  3️⃣  Sensitive Data Masking Layer (SER-AP-003)              │
│      → 입력 필터링 (Inbound) + 출력 필터링 (Outbound)       │
├─────────────────────────────────────────────────────────────┤
│  4️⃣  Agent Secure Coding Layer (SER-AP-004)                 │
│      → 프롬프트 캡슐화 + 제로 트러스트 + 감시 로깅            │
├─────────────────────────────────────────────────────────────┤
│  ⭐ Hybrid Multi-LLM Orchestration (아키텍처)                │
│      → SaaS API (성능) + Private LLM (민감정보) 자동 라우팅  │
└─────────────────────────────────────────────────────────────┘
```

## 🏗️ Part 3: 상세 기술 설계 (실전 개발 관점)

### 3.1 입출력 양방향 데이터 정제 레이어 (SER-AP-001, 003)

**개념:** LLM으로 들어가는 모든 데이터와 나오는 답변을 양방향으로 검증·마스킹하는 중간 인터셉터 구조

**구현 구조:**

```
User Input
  ↓
[Inbound Interceptor - 입력 필터]
  ├─ 민감정보 자동 탐지 & 마스킹 (PII Detection Engine)
  ├─ 입력값 검증 & 클렌징
  └─ 프롬프트 인젝션 시그니처 차단
  ↓
[LLM Model]
  ↓
[Outbound Interceptor - 출력 필터]
  ├─ 응답 내 민감정보 재차 마스킹
  ├─ 악의적 스크립트 탐지
  └─ KITA 데이터 외부 유출 조항 검증
  ↓
User Output
```

**구현 기술:**
- **PII Detection:** NER(Named Entity Recognition) 기반 또는 정규표현식 패턴
- **프롬프트 인젝션 방어:** 시스템 프롬프트를 사용자 입력과 물리적으로 분리하는 캡슐화
- **마스킹 전략:** `[PII-숫자-임시ID]` 방식의 토큰화 (원본 추적 가능하게)

---

### 3.2 데이터 접근 제어 (SER-AP-002) - RBAC + ABAC 듀얼 레이어

#### **1차 방어선: RBAC (Role-Based Access Control)**
**대상:** 서비스 메뉴, 에이전트 Tool 실행 권한, API 게이트웨이 엔드포인트

**설정 예시:**
- Admin: 모든 에이전트, 모든 데이터 접근 가능
- 관세사: 통관 대행, 관세 환산 에이전트만 실행 가능
- 일반 회원: 조회 전용 에이전트만 실행 가능

#### **2차 방어선: ABAC (Attribute-Based Access Control)**
**대상:** 데이터 자산(DB 행, 파일, RAG 컨텍스트)의 실제 접근

**동작 원리:**
```
에이전트가 쿼리 발생
  ↓
[ABAC Policy Engine (OPA/PBAC)]
  ├─ User Attribute: Company_ID = 'KITA_001'
  ├─ Data Attribute: Owner_Company_ID = 'KITA_001'
  └─ Environment: Data_Classification <= 'Level_2'
  ↓
조건 일치 → 데이터 반환 / 불일치 → 접근 차단 또는 마스킹
```

**개발 관점 - 복잡도 낮추기:**

| 데이터 형태 | 필터링 방식 | 구현 기술 |
|-----------|-----------|---------|
| **구조화 (DB/RDB)** | 동적 WHERE 절 주입 | Spring Data JPA Interceptor + WHERE `company_id = :userCompanyId` |
| **반구조 (벡터 DB)** | 메타데이터 필터링 | Qdrant Payload Filter or PostgreSQL WHERE 절 |
| **비구조 (파일)** | 메타데이터 태깅 | Object Storage Tagging (S3) + 메타데이터 검증 |

**핵심 원칙:** 비즈니스 로직에 권한 체크를 섞지 말고, **중앙 보안 엔진(Policy Engine)**이 모든 쿼리/접근에 자동으로 필터링 조건을 끼워 넣도록 설계.

### 3.3 Agent 시큐어 코딩 (SER-AP-004)

#### **프롬프트 인젝션 방어 아키텍처**

```
[System Prompt - 격리된 영역]
  "You are a trade assistant. Only use provided tools..."
  
[User Input - 격리된 영역]
  User 질문
  
[Tool Definitions - 격리된 영역]
  - Tool A: 수출입 통계 조회
  - Tool B: 관세율 계산
  
[LLM]
  → 프롬프트 인젝션 패턴 사전 필터링
  → 요청된 Tool만 선택 (다른 Tool 호출 차단)
```

#### **Zero Trust + 최소 권한 (API 호출 제어)**

| 단계 | 메커니즘 |
|-----|---------|
| 1️⃣ **인증** | 에이전트가 Tool을 호출할 때마다 Vault에서 동적으로 인증서/토큰 발급 |
| 2️⃣ **인가** | 특정 Tool에 할당된 최소 권한의 API 키만 제공 (일반 관리자 키 ❌) |
| 3️⃣ **감시** | 모든 API 호출을 타임스탬프 + 컨텍스트와 함께 중앙 로그에 기록 |

**구현 솔루션:**
- **Vault:** HashiCorp Vault (온프레미스) 또는 AWS Secrets Manager
- **API Gateway:** Kong, AWS API Gateway, 또는 커스텀 Proxy
- **Logging:** ELK Stack (Elasticsearch + Logstash + Kibana) 또는 Datadog

### 3.4 하이브리드 Multi-LLM 오케스트레이션

**배경:** RFP는 온프레미스 vs SaaS를 특정하지 않았으므로, 우리는 두 장점을 결합한 하이브리드를 제안.

**아키텍처:**

```
User Request
  ↓
[AI Sandbox - Request Classifier]
  ├─ 민감정보 포함 여부 분석
  ├─ 필요한 연산 복잡도 판단
  └─ 데이터 접근 범위 검증
  ↓
  ├─ (민감정보 없음 + 복잡한 문맥)
  │  → SaaS API (OpenAI GPT-4, Claude 등)로 라우팅
  │     "데이터 학습 금지" 옵션 활성화
  │
  └─ (민감정보 포함 or 폐쇄망)
     → Private LLM (Llama, Mistral 등)로 라우팅
        클라우드 독립 보안 존에서 로컬 실행

[응답 처리]
  → 출력 필터 적용 (민감정보 마스킹)
  → 사용자에게 반환
```

**기대 효과:**
- 성능: SaaS의 고급 모델 활용
- 보안: 민감 데이터는 절대 외부로 나감 ❌
- 비용: 필요한 부분만 SaaS 사용

## 📊 Part 4: Vector DB 선택 기준

### 4.1 추천 선택지

| DB | 특징 | ABAC 메타데이터 필터링 | 이기종 데이터 지원 | 추천 상황 |
|----|----|-----------------|-----------------|---------|
| **Qdrant** ⭐ | 초고속 Rust 기반 | 🟢 우수 (Payload Filter) | 🟡 중간 | 성능 우선, 보안 강화 필요 |
| **PostgreSQL+pgvector** ⭐ | 기존 RDBMS 활용 | 🟢 우수 (SQL WHERE) | 🟢 우수 | 개발 복잡도 낮추기, 메타데이터 풍부 |
| **Milvus** | 대규모 분산 | 🟢 우수 | 🟡 중간 | 데이터 규모 매우 큼 |
| **OpenSearch/Elasticsearch** | 문서 검색 강점 | 🟡 중간 (DLS) | 🟢 우수 | 비구조 파일 + 벡터 혼합 |
| Pinecone | SaaS 관리형 | 🟢 우수 | 🟡 중간 | 인프라 관리 최소화 (데이터 유출 조항 확인 필수) |

### 4.2 KITA 사업에 대한 최종 추천

**시나리오 A: "개발 빠르고 유지보수 쉽게" (안정형)**
```
PostgreSQL + pgvector
├─ 기존 사용자 정보, 권한 메타데이터와 같은 테이블에서 관리
├─ ABAC 필터링이 표준 SQL WHERE로 처리됨
└─ 개발자가 권한 체크 로직을 따로 짤 필요 없음
```

**시나리오 B: "AI 전문성 강조 + 고성능" (성능형)**
```
Qdrant (On-Premise)
├─ SeekersLab KYRA의 차별화된 고속 벡터 처리 강조
├─ 클라우드 독립 배포로 데이터 폐쇄성 강조
└─ 보안 감시 레이어(Inbound/Outbound Filter)와 찰떡궁합
```

**권장:** **PostgreSQL + pgvector** (개발 복잡도 낮추고, 동시에 ABAC 메타데이터 필터링을 우아하게 처리)
